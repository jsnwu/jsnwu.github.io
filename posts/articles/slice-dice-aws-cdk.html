Shipping Next.js & Django Web App to ECS Fargate

Apr 2, 2026 #AWS #CDK #ECS #Next.js #Django 5 min

Intro

Created a web app Slice Dice delivery recently, it works perfectly locally inside docker containers. Deploying to a remote enviroment will be the next step. AWS ECS will be a good choice since I've been using it on daily basis at work. AWS CDK can make the task a breeze, the infrastructure as code is the way to go to avoid all manual configurations and headaches. AWS CDK is similar to Terraform, but has typescript support. Instead of learning a new language(HCL), I can use my favorite language to write the infrastructure code with less friction. The app is has all basic features like menu, cart, checkout, track order, and owner dashboard for user, delivery, and inventory management. One service runs two containers: Next.js on port 3000 and Django on 8000. The ALB sends /api/* to the API and everything else to the frontend—no extra reverse proxy in the cluster.

Before you deploy

You need Node/npm for the CDK package, AWS credentials (aws configure or SSO), and a one-time npx cdk bootstrap per account and region. Optional settings for the stack are loaded from .env when present. 

              
                npm install -g aws-cdk
                aws configure
                npx cdk bootstrap
                npm install

Images and ECR

The stack references ECR repos named slice-dice-frontend and slice-dice-backend but does not create them, so redeploys are not blocked by “repository already exists.” Create the repos once if needed, or from the repository root run the bundled ECR script that lives under infra (optionally with region and tags). From the same package directory where you run npm install, npm run ecr-deploy does the same job.

First stack deploy

In the infra directory that contains the stack’s package.json, run npm install, then bash scripts/deploy.sh (ensures the ECS log group exists, builds, and runs cdk deploy) or invoke npx cdk deploy SliceDiceFargateStack directly. CloudFormation outputs worth noting include AlbDnsName, AlbHttpsUrl when TLS is wired, cluster and service names for later operations, and the ECR URIs.

Postgres RDS

The backend expects standard Django/Postgres variables (POSTGRES_DB, POSTGRES_USER, DJANGO_SECRET_KEY, and either in-VPC RDS from the stack or an external host plus password/secret). To let CDK provision a small RDS instance, use npm run rds-deploy, set CDK_PROVISION_RDS=true in infra/.env, or run CDK_PROVISION_RDS=true bash scripts/deploy.sh. For an external database, leave that unset and configure POSTGRES_HOST and credentials in env or Secrets Manager as documented in the project.

Once the DB is reachable from the tasks, bash scripts/run-migrate-seed.sh runs Django migrate and seed_demo as a one-off Fargate task; add --migrate-only to skip seeding. Backup retention defaults to 0 for Free Tier–friendly RDS; raise RDS_BACKUP_RETENTION_DAYS on a paid account when you want real retention.

Healthy behind the load balancer

Once tasks are running, the ECS service view ties each container port to an ALB target group. Both the Next.js task (frontend:3000) and the Django API (backend:8000) should register as healthy; CPU and memory charts stay modest for a quiet workload.

The App is live

Once the deployment is completed and the health check is green, the app is live and ready to use.

Shipping new container builds

After you push new :latest images, npm run ecs-force or bash scripts/deploy-ecs-force.sh forces ECS to roll tasks and pull fresh images without touching CloudFormation. If you change CPU, memory, environment, or container definitions, you need a full cdk deploy / deploy.sh again—not only a force rollout.

When a deploy fails

A stack stuck in ROLLBACK_COMPLETE has to be deleted in CloudFormation (or via a cleanup script such as scripts/delete-ecs.sh, after confirming stack name and region) before you can deploy again.
